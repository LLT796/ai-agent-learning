# Day 5 开发日志：Agent 记忆模块

> 日期：2026-04-03
> 目标：给 Agent 加上短期记忆、长期记忆和工作记忆

---

## 一、为什么 Agent 需要记忆

没有记忆的 Agent 每一轮对话都是"失忆"的。用户说"帮我看看刚才那个耳机"，Agent 完全不知道"刚才那个"是什么。

记忆解决的核心问题是**上下文连续性**——让 Agent 在多轮交互中表现得像一个记得住事情的人，而不是每次都从零开始。

但记忆不是越多越好。LLM 的上下文窗口有限，把所有历史对话塞进去会超限；无关信息太多还会干扰 LLM 的判断。所以需要不同类型的记忆机制，分别解决不同场景的问题。

---

## 二、三种记忆的定位

```
短期记忆：你正在和客户聊天时，脑子里记着这次对话说了什么
长期记忆：你记得这个客户上周来过，买了一双跑鞋
工作记忆：你在草稿纸上记着"客户预算1500，偏好国际品牌，已看过索尼和华为"
```

| 维度 | 短期记忆 | 长期记忆 | 工作记忆 |
|------|----------|----------|----------|
| 存什么 | 当前对话的消息历史 | 历史会话的摘要 | 推理过程中的结构化笔记 |
| 存在哪 | messages 列表（内存） | ChromaDB（磁盘） | State 字段（内存） |
| 生命周期 | 当前会话 | 永久（跨会话） | 单次推理任务 |
| 谁写入 | 自动累积 | 会话结束时保存 | Agent 或代码逻辑主动写入 |
| 谁读取 | LLM 每轮推理都读 | 新会话开始时按相关性召回 | 注入到 System Prompt |
| 解决什么问题 | "刚才说的那个" | "上次看的那个" | "用户要什么我整理一下" |

## 三、短期记忆：三种策略详解

### 3.1 问题

多轮对话中，messages 列表不断增长。一轮对话约产生 4 条消息（Human + AI(tool_calls) + Tool + AI(answer)），聊 10 轮就是 40 条消息。加上工具返回的长文本，很快就超过 LLM 的上下文窗口。

### 3.2 策略一：滑动窗口

```python
def sliding_window(messages, max_pairs=10):
    # 只保留最近 max_pairs 轮的消息，早期消息直接丢弃
```

**原理**：像一个固定大小的窗口在消息列表上滑动，窗口外的消息直接丢掉。

**优点**：实现最简单，性能最好（O(1) 操作），不需要额外 API 调用。

**缺点**：早期信息完全丢失。如果用户在第 2 轮说了预算 1000，到第 12 轮时这条信息就没了。用户说"我最开始说的预算是多少来着"，Agent 答不上来。

**适用场景**：简单的问答场景，对话不会太长，或者早期信息确实不重要。

### 3.3 策略二：Token 截断

```python
def token_trim(messages, max_tokens=4000):
    # 按 Token 总数控制，从最早的消息开始删，直到总量低于阈值
```

**原理**：精确计算所有消息的 Token 数，超过限制时从最早的消息开始移除。

**优点**：精确控制上下文长度，不会超过 LLM 限制。

**缺点**：可能在一轮对话的中间切断（比如保留了 AI 的工具调用，但丢了工具的返回值），导致 LLM 看到不完整的信息链。`allow_partial=False` 可以避免切断单条消息，但解决不了切断对话轮次的问题。

**适用场景**：需要严格控制上下文长度时，常与其他策略组合使用。

### 3.4 策略三：摘要压缩

```python
def summarize_and_trim(messages, max_pairs=6, summary_threshold=12):
    # 早期对话 → 压缩成一段摘要
    # 近期对话 → 保留原文
    # 摘要作为 SystemMessage 插入
```

**原理**：把早期对话压缩成一段摘要（"用户问了跑鞋，我推荐了 Nike Pegasus"），只保留近期对话的原文。摘要插入到消息列表开头，LLM 既能看到早期的概要信息，又能看到近期的完整细节。

**优点**：不完全丢失早期信息，在信息保留和长度控制之间取得平衡。

**缺点**：
- 摘要会损失细节（具体参数、价格等可能被丢掉）
- 如果用 LLM 做摘要，需要额外的 API 调用（多花 Token 和时间）
- 本项目用了简单的文本提取代替 LLM 摘要，效果不如真正的 LLM 摘要但省成本

**适用场景**：长对话、多轮交互、需要保留早期上下文概要的场景。**生产环境推荐方案**。

### 3.5 三种策略选择指南

```
对话 < 10 轮           → 不需要任何策略，直接传全量 messages
对话 10-20 轮          → 滑动窗口（简单场景）或摘要压缩（复杂场景）
对话 > 20 轮           → 必须用摘要压缩
需要严格控制 Token 消耗 → Token 截断 + 滑动窗口组合
生产环境               → 摘要压缩 + Token 截断双重保险
```

## 四、长期记忆：跨会话的"经验"

```python
        # 在 ChromaDB 中找最相似的历史摘要
        # 返回 top-K 条历史记录
```

本质上就是 Day 2 RAG 的应用——只不过存的不是文档，而是历史对话摘要。检索时用当前问题做语义搜索，找到相关的过去交互。

### 4.2 写入时机

每次会话结束时调用 `save_conversation()`，把这次对话的精华存下来。不是存原始的 40 条消息，而是提取后的摘要（"用户咨询了降噪耳机，推荐了 Sony XM5 和华为 FreeBuds Pro 3"）。

### 4.3 读取时机

新会话开始时，用用户的第一个问题做一次召回。如果命中了相关历史，把历史信息注入到 System Prompt 或工作记忆中。

```python
memories = long_term.recall("上次看的那个鞋子还在卖吗", k=2)
# → 召回: "用户问了跑鞋，推荐了 Nike Pegasus 41，¥899"
```

### 4.4 适用场景

- 电商导购：记住用户的购物偏好和历史咨询
- 客服系统：记住用户之前的工单和解决方案
- 个人助理：记住用户的习惯和常用设置

### 4.5 注意事项

- 存储量会持续增长，需要定期清理过期记录
- 召回的历史可能过时（上次推荐的商品可能已下架）
- 隐私问题：用户数据持久化需要合规处理

## 五、工作记忆：推理中的"草稿纸"

### 5.1 实现方式

```python
class WorkingMemory:
    notes: dict           # 自由笔记
    user_preferences: dict  # 提取的用户偏好
    collected_products: list  # 已查看的商品 ID

    def to_context_string(self):
        # 把所有信息拼成文本，注入到 Prompt
        # → 【用户偏好】预算: 1500, 用途: 跑步
        # → 【已查看商品】SKU001, SKU003
```

### 5.2 和短期记忆的区别

短期记忆是**原始的消息流**——所有 Human/AI/Tool 消息的完整记录。LLM 要从一大堆消息里自己提取有用信息。

工作记忆是**结构化的提取结果**——已经整理好的用户偏好、已查看的商品、中间结论。LLM 一看就知道用户要什么。

类比：短期记忆是你的聊天记录，工作记忆是你边聊边在便签纸上写的要点。

### 5.3 注入方式

工作记忆转成文本后，追加到 System Prompt 末尾：

```python
enhanced_prompt = SHOPPING_AGENT_PROMPT + f"\n\n## 当前工作记忆\n{wm_context}"
```

LLM 每轮推理时都能看到这些结构化信息，不需要从历史消息中自己翻找。

### 5.4 实验观察

同样问"还有什么推荐"：
- 无工作记忆：Agent 追问"请问您想找什么类型的商品？预算多少？"
- 有工作记忆（预算1500、用途跑步、品牌国际）：Agent 直接搜索匹配的商品并推荐

工作记忆的价值在于**减少 LLM 的信息提取负担**，让它专注于推理和决策。

### 5.5 适用场景

- 复杂导购：用户需求分多轮逐步明确时
- 多步任务：Agent 需要跨多个工具调用累积信息时
- 表单填写：逐步收集信息直到完整

## 六、三种记忆在 Agent 中的协作

一次完整的导购对话中，三种记忆同时工作：

```
新会话开始
  ↓
长期记忆：召回用户的历史交互（"上次咨询过跑鞋"）
  ↓
注入到 System Prompt
  ↓
用户第 1 轮：我想买个耳机
  → 工作记忆记录：偏好=耳机
  → 短期记忆：累积消息
  ↓
用户第 2 轮：预算 1500 以内
  → 工作记忆更新：预算=1500
  → 短期记忆：累积消息
  ↓
用户第 3 轮：帮我看看索尼那款
  → Agent 用工作记忆（品类+预算）+ 短期记忆（"索尼"）来搜索
  ↓
...
  ↓
用户第 10 轮：消息超限了
  → 短期记忆：摘要压缩早期对话，保留近期
  → 工作记忆：仍然完整保留所有偏好信息
  ↓
会话结束
  → 长期记忆：存储本次对话摘要
```

关键洞察：**短期记忆是时间维度的（最近的对话），工作记忆是结构维度的（整理后的要点），长期记忆是经验维度的（历史积累）。三者互补，不冲突。**

## 八、今日踩过的坑

1. **消息列表的引用问题**：在 `agent_node` 中直接修改 `state["messages"]` 会影响原始数据。必须用 `list(state["messages"])` 做浅拷贝后再操作。

2. **ChromaDB 重复初始化**：`LongTermMemory` 每次实例化都连接 ChromaDB。如果在循环中多次创建实例，可能产生连接泄露。应该全局初始化一次。

3. **摘要压缩的质量权衡**：用 LLM 做摘要效果最好但成本高（每轮都要额外调一次 API）。本项目用了简单的文本提取（取 HumanMessage 和 AIMessage 的前 100 字符），效果够用且零额外成本。生产环境可以按需切换为 LLM 摘要。

4. **工作记忆的写入时机**：当前实现是在代码中手动写入 `working_mem.set_preference()`。更理想的方式是让 Agent 自己决定什么时候写（通过一个专门的"记笔记"工具），这个可以后续优化。
