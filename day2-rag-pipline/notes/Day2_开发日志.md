# Day 2 开发日志：RAG Pipeline、Chunking 策略与 Retriever 工作方式

> 日期：2026-03-31
> 目标：跑通 文档加载 → 分块 → Embedding → 向量存储 → 检索 → 生成 完整链路

---

## 一、RAG 的核心思想

RAG（Retrieval-Augmented Generation）解决的是一个根本问题：**LLM 不知道你的私有数据**。

LLM 的训练数据是公开互联网内容，它不知道你的商品库有什么、价格多少、退换货政策是什么。直接问它，它只能编造。RAG 的做法是：先从你的知识库里检索出相关内容，再把这些内容作为上下文喂给 LLM，让它"基于事实"回答。

本质上 RAG 就是一个**开卷考试**——LLM 本身很聪明但不知道答案，你把参考资料翻到相关页递给它，它就能给出准确回答。

---

## 二、RAG Pipeline 完整数据流

```
离线索引阶段（只跑一次）：
  原始文档（PDF/TXT）
    → DocumentLoader 加载为文本
    → TextSplitter 切分为小块（chunks）
    → Embedding Model 把每个块转为向量（如 1024 维浮点数组）
    → VectorStore（ChromaDB）存储向量 + 原文

在线查询阶段（每次提问）：
  用户问题
    → 同一个 Embedding Model 把问题转为向量
    → VectorStore 做相似度搜索，返回 top-K 个最相关的块
    → 把检索到的块 + 用户问题拼接为 Prompt
    → LLM 基于上下文生成答案
```

关键洞察：**索引阶段和查询阶段必须使用同一个 Embedding 模型**，否则向量空间不一致，检索结果会错乱。

## 三、文档加载（Document Loading）

### 3.1 两种加载器

| 加载器 | 用途 | 安装依赖 |
|--------|------|----------|
| `TextLoader` | 加载 .txt 纯文本文件 | `langchain-community` |
| `PyPDFLoader` | 加载 .pdf 文件，按页拆分 | `langchain-community` + `pypdf` |

### 3.2 加载器返回的数据结构

加载器返回 `Document` 对象的列表，每个 Document 包含：

```python
Document(
    page_content="文档的文本内容...",
    metadata={"source": "product_knowledge.txt", "page": 0}
)
```

`metadata` 很重要——后续检索时可以用它追溯答案来源于哪个文档、哪一页。

## 四、文本分块策略（Chunking）—— 最核心的工程决策

### 4.1 为什么必须分块

三个原因：

1. **LLM 上下文窗口有限**：不能把整个知识库塞进去
2. **检索精度**：用户问"耳机降噪效果"，我需要精确返回描述降噪的那个段落，而不是整篇文档
3. **向量表示质量**：一段太长的文本转成单个向量，语义会被"稀释"，检索效果变差

### 4.2 RecursiveCharacterTextSplitter 的工作方式

它按照一个**优先级列表**依次尝试分割：

```python
separators=["\n\n", "\n", "。", "！", "？", "；", " ", ""]
```

逻辑是：
1. 先尝试按 `\n\n`（段落）分割
2. 如果某个段落还是超过 chunk_size，再按 `\n`（换行）分割
3. 还超的话按句号分割
4. 以此类推，直到每个块都不超过 chunk_size

这比简单地按固定字符数截断要好得多——它尽量保持语义完整性。

### 4.3 chunk_size 的取舍（实验结论）

| chunk_size | 块数量 | 优点 | 缺点 | 实验观察 |
|------------|--------|------|------|----------|
| 300 | 很多 | 检索精度高，定位准确 | 信息碎片化，上下文不完整 | 降噪和续航信息被切到不同块，只检索到其中一个 |
| 500 | 适中 | 一个块≈一个完整知识点 | 平衡点，需微调 | 一个块基本包含一个商品的某个方面完整描述 |
| 1000 | 较少 | 上下文丰富，信息完整 | 混入无关内容，精度下降 | 一个块可能同时包含降噪和外观描述，检索时无关内容稀释相关性 |
| 2000 | 很少 | 几乎不丢失信息 | 接近全文检索，失去精确定位优势 | 一个块包含大半个商品描述，向量表示模糊 |

**结论：中文电商场景推荐 400-600 字符。** 核心原则是让一个 chunk 刚好包含一个完整的知识点。

### 4.4 chunk_overlap 的取舍（实验结论）

overlap 是相邻块之间的重叠区域，目的是**防止关键信息在分割边界被切断**。

| overlap | 效果 | 实验观察 |
|---------|------|----------|
| 0 | 块之间完全不重叠 | "降噪深度35dB"如果正好在切割点，前一个块只有"降噪深度"，后一个块只有"35dB" |
| 50 | 少量重叠 | 部分缓解边界问题，但仍有遗漏 |
| 100 | 推荐值（chunk_size的20%） | 边界信息基本能在至少一个块中保持完整 |
| 200 | 过度重叠（40%） | 检索结果中出现高度相似的重复块，浪费 top-K 名额 |

**结论：overlap 设为 chunk_size 的 15%-25%。** 太小会丢边界信息，太大会导致冗余。

### 4.5 分块的"黄金法则"

> 一个 chunk 应该能独立回答一个问题。

如果你拿到一个 chunk 读一遍，发现它是半截话、缺乏上下文、看不懂在说什么——那就说明分块策略有问题。

## 五、Embedding（文本向量化）

### 5.1 Embedding 做了什么

把一段文本转成一个固定长度的数字数组（向量），使得**语义相近的文本在向量空间中距离近**。

```
"Sony降噪耳机续航8小时" → [0.12, -0.34, 0.56, ..., 0.78]  # 1024 维
"索尼无线耳机电池能用多久" → [0.11, -0.33, 0.55, ..., 0.77]  # 很接近！
"耐克跑鞋透气网面" → [-0.45, 0.67, -0.23, ..., 0.12]  # 差别很大
```

关键理解：Embedding 捕捉的是**语义**，不是字面。"续航8小时"和"电池能用多久"字面完全不同，但向量非常接近。

### 5.2 本项目使用的 Embedding 模型

通过通义千问的 OpenAI 兼容接口调用：

```python
embeddings = OpenAIEmbeddings(
    api_key=DASHSCOPE_API_KEY,
    base_url="https://dashscope.aliyuncs.com/compatible-mode/v1",
    model="text-embedding-v3",
)
```

和调用 LLM 一样，改 `base_url` 和 `model` 就行，代码层面完全兼容 OpenAI 的接口。

## 六、向量存储（Vector Store）—— ChromaDB

### 6.1 ChromaDB 的角色

ChromaDB 是一个轻量级向量数据库，做两件事：

1. **存储**：保存每个 chunk 的向量和原始文本
2. **检索**：给定一个查询向量，快速找到最相似的 K 个向量（及其对应的文本）

### 6.2 存储结构

```
ChromaDB 中每条记录：
├── id: 唯一标识
├── embedding: [0.12, -0.34, ...] （向量，1024维）
├── document: "Sony WF-1000XM5 降噪耳机..." （原始文本）
└── metadata: {"source": "product_knowledge.txt"} （元数据）
```

### 6.3 为什么选 ChromaDB

学习阶段用它因为：零配置、纯 Python、支持持久化到本地文件。生产环境可能会换成 Milvus、Pinecone、Weaviate 等，但接口层面（LangChain 的 VectorStore 抽象）是一样的，换数据库只改初始化代码。

## 七、Retriever 的工作方式

### 7.1 检索流程

```python
retriever = vectorstore.as_retriever(
    search_type="similarity",  # 余弦相似度
    search_kwargs={"k": 3},    # 返回 top-3
)
results = retriever.invoke("耳机降噪效果怎么样？")
```

内部发生的事情：

1. 调用 Embedding 模型，把 "耳机降噪效果怎么样？" 转成向量 Q
2. 在 ChromaDB 中计算 Q 与所有存储向量的余弦相似度
3. 按相似度从高到低排序，取前 K 个
4. 返回这 K 个向量对应的原始文本块

### 7.2 余弦相似度（Cosine Similarity）

```
相似度 = cos(θ) = (A · B) / (|A| × |B|)

值范围：-1 到 1
  1  = 完全相同方向（语义一致）
  0  = 正交（语义无关）
  -1 = 完全相反方向（语义相反）
```

实际检索中，大部分结果的相似度在 0.3-0.9 之间。通常 > 0.7 算是比较好的匹配。

### 7.3 K 值的选择

| K 值 | 效果 |
|------|------|
| K=1 | 只取最相关的一个块。如果答案分散在多个块中，会遗漏 |
| K=3 | 推荐值。大多数情况下 3 个块能覆盖答案所需信息 |
| K=5 | 信息更全，但可能引入无关块，干扰 LLM 判断 |
| K=10 | 几乎全文检索，失去 RAG 的意义，且消耗更多 Token |

**K 值也不是越大越好**——无关块混进去后，LLM 可能被误导。

### 7.4 检索类型对比

| 类型 | 原理 | 适用场景 |
|------|------|----------|
| `similarity` | 纯余弦相似度 | 通用场景，默认选择 |
| `mmr` (Max Marginal Relevance) | 在相关性和多样性之间平衡 | 防止返回的 K 个块内容高度重复 |
| `similarity_score_threshold` | 只返回相似度高于阈值的块 | 需要严格控制检索质量时 |

本项目使用的是 `similarity`，最简单直接。当你发现检索结果中有太多重复内容时，可以考虑切换到 `mmr`。

## 八、RAG Prompt 的设计

```python
RAG_PROMPT = """你是淘天自营平台的智能客服助手。
请根据以下商品知识库中的信息来回答用户的问题。

要求：
1. 只基于提供的上下文信息回答，不要编造数据
2. 如果上下文中没有相关信息，诚实地说"根据现有资料无法回答"
3. 回答要具体，包含价格、参数等关键数据

===== 知识库相关内容 =====
{context}
===== 知识库结束 =====

用户问题：{question}
"""
```

设计要点：

1. **明确限制 LLM 只用上下文回答**：防止它用训练数据编造
2. **要求诚实说不知道**：比胡编乱造好得多
3. **用分隔符包裹上下文**：让 LLM 清楚区分上下文和指令
4. **context 和 question 分开放**：LLM 能清楚区分"参考资料"和"要回答的问题"

## 九、实验中的关键发现

### 9.1 精确问题 vs 模糊问题

- **精确问题**（"XM5 续航多久"）：检索效果很好，top-1 就能命中
- **模糊问题**（"跑步用什么鞋好"）：能检索到跑鞋相关内容，但可能遗漏某些商品
- **超范围问题**（"MacBook 多少钱"）：检索仍会返回结果（最"不远"的），但 LLM 能正确判断上下文中没有答案

### 9.2 对比问题的挑战

问"索尼和华为耳机哪个好"时，需要同时检索到两个品牌的信息。如果 K=3 但只检索到索尼的三个块，回答就会片面。解决方案：增大 K 值，或者在查询前做意图分析、拆成两个子查询。

### 9.3 分块策略没有银弹

最重要的认知：**不存在万能的分块参数**。最佳策略取决于：
- 文档的结构（有没有清晰的章节划分）
- 典型的用户问题（是问单个知识点还是对比多个）
- 知识点的长度分布（一个商品描述通常多少字）

必须用真实问题测试、迭代优化，不能凭感觉定参数。

这就是 Agentic RAG——Agent + RAG 的组合。
